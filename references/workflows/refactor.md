# Refactor 워크플로우 (리팩토링)

**목적**: 외부 동작은 그대로 유지하고 내부 구조를 개선.

**핵심 질문**: "동작은 그대로 두고 구조를 더 낫게 만들 수 있는가?"

목표는 "예쁜 코드" 가 아니라 **변경하기 쉬운 코드**. 테스트 없이 리팩토링하면 사실상 재구현.

---

## Phase 템플릿

1. **현재 동작 테스트로 고정** — 변경 전 테스트가 통과하는지 확인. 커버리지가 약하면 **먼저 characterization test 추가** (현재 동작을 검증하는 테스트 — 옳든 그르든 그대로 박제). 이게 안전망.
2. **개선 목표 명시** — 다음 중 어떤 것인지 명확히:
   - 중복 제거
   - 책임 분리 (god class / god function 쪼개기)
   - 가독성 (네이밍 / 구조 / 흐름)
   - 확장성 (새 케이스 추가가 쉽도록)
   - 성능 (단, 측정 기반)
   "왜 지금 필요한가?" 도 명시 — 다음 기능 작업이 막혔다 / PR 리뷰에서 반복 지적됐다 등.
3. **작은 단위 변경** — 각 변경은 **한 가지 개선만**. 큰 리팩토링은 시리즈 PR 로 분할. 한 PR 에 "중복 제거 + 책임 분리 + 네이밍" 섞지 말 것.
4. **매 변경마다 테스트** — 1단계 테스트가 그대로 통과해야 함. **떨어지면 즉시 되돌리기** — 디버깅으로 시간 쓰지 말고 다른 접근.
5. **사이드이펙트 확인** — 성능 회귀 / 메모리 / 호출 빈도 / 로그 양 변화 없는지. 변경 전후 메트릭 비교.
6. **불필요한 추상화 정리** — 새로 만들기보다 **있는 추상화도 의심**. 사용처 1곳뿐이면 인라인이 답. "혹시 나중에 …" 로 만들어진 인터페이스 / 팩토리는 제거 후보.
7. **코드 리뷰** — 리뷰어는 "변경 후 코드만 봐도 의도가 명확한가" 기준으로 본다. 변경 이력을 모른다고 가정하고 신규 코드만 평가.
8. **마무리 (모든 타입 공통)** — `/orch:report <issue_id>` 호출 (REPORT-data.md / REPORT.html + errors_check / ai_ready_check 후보 도출 + orch 인박스로 `[follow-up-candidates]` 송신) → `/orch:issue-down <issue_id>` 로 cascade shutdown. orch 가 `/orch:report` 자동 호출하지 않으므로 leader 가 이 단계를 빠뜨리면 REPORT.html 누락.

---

## Review 체크리스트 (reviewer 용)

reviewer 가 PR 을 평가할 때 다음 항목을 우선:

1. **외부 동작 보존** — public API / 파일 포맷 / DB 스키마 / 로그 메시지 등 외부 인터페이스가 그대로인가? 변경됐다면 그건 리팩토링이 아니라 **변경** 이므로 별도 PR.
2. **테스트 무수정** — 기존 테스트의 assertion 이 바뀌었다면 외부 동작이 바뀐 것 — 의도된 것인지 확인. characterization test 가 추가됐는가?
3. **기능 변경 / 버그 픽스 혼입** — diff 에 리팩토링이 아닌 변경 (새 기능 / 버그 fix) 이 섞여 있지 않은가? 섞여 있으면 분리 요구.
4. **추상화 적정성** — 새로 만든 추상화 (인터페이스 / 헬퍼 / 클래스) 의 사용처가 2곳 이상인가? 1곳뿐이면 과한 추상화.
5. **사이드이펙트** — 성능 / 메모리 / 호출 빈도 변화 없는가? PR 본문에 측정 결과 있는가?
6. **diff 크기** — 너무 크면 시리즈로 분할 요구. 한 PR 에 여러 종류 리팩토링이 섞이면 회수 / 진단 비용이 큼.

---

## 주의

- 리팩토링 중 **기능 변경 / 버그 픽스 섞기 금지** — 별도 PR 로 분리.
- 테스트 없이 리팩토링하면 사실상 재구현 — 위험 신호.
- 외부 인터페이스 변경은 리팩토링이 아니라 **변경** — 워크플로우 다시 선택해야 함.
- "예쁜 코드" 가 목표 아님 — **변경 쉬움** 이 목표.
