# Feature 워크플로우 (기능 구현)

**목적**: 새로운 동작 / 기능을 추가.

**핵심 질문**: "무엇이 가능해져야 하는가?"

코드보다 **문제 정의와 인터페이스 설계** 가 더 중요. 요구사항이 흔들리면 구현이 흔들린다.

---

## Phase 템플릿

1. **요구사항 정의** — 유스케이스, 입력, 출력, 비기능 요건 (성능 / 보안 / 호환성) 을 문장으로 정리. 모호하면 여기서 `AskUserQuestion` 또는 orch escalate — 추측 진행 금지.
2. **인터페이스 설계** — 데이터 모델 / API 시그니처 / 상태 변화 / 호환성 영향. 큰 기능이면 PM 워커 spawn 해 설계 산출물 (`docs/spec/<id>/...`) 부터 받고 사용자 컨펌.
3. **스코프 분할** — 작업 가능한 단위로 쪼개기. 각 단위는 **독립적으로 머지 가능** 해야 함 (한 PR 이 다른 PR 의 미머지 코드에 의존하면 phase 분리 실패).
4. **테스트 케이스 정의** — 정상 케이스 + 엣지 케이스 + 실패 모드. 가능하면 TDD (테스트 먼저 작성 → 빨강 확인 → 구현 → 초록).
5. **구현** — 분할 단위 순차. 각 단위마다 테스트 통과 + 자체 PR.
6. **검증** — 정상/예외 케이스 수동 확인, 기존 회귀 테스트 그린, UI 변경이면 브라우저로 사용해 보고 보고.
7. **문서화 / 배포** — README / CHANGELOG / docs 업데이트. 배포 후 모니터링 포인트 (어떤 메트릭·로그를 봐야 회귀 감지?) 정의.
8. **마무리 (모든 타입 공통)** — `/orch:report <issue_id>` 호출 (REPORT-data.md / REPORT.html + errors_check / ai_ready_check 후보 도출 + orch 인박스로 `[follow-up-candidates]` 송신) → `/orch:issue-down <issue_id>` 로 cascade shutdown. orch 가 `/orch:report` 자동 호출하지 않으므로 leader 가 이 단계를 빠뜨리면 REPORT.html 누락.

---

## Review 체크리스트 (reviewer 용)

reviewer 가 PR 을 평가할 때 다음 항목을 우선:

1. **인터페이스** — public API / 데이터 모델이 요구사항을 정확히 표현하는가? 호환성을 깨는가?
2. **엣지 케이스** — empty / null / 경계값 / 동시성 / 권한 경계 누락?
3. **테스트 커버리지** — 정상 케이스만 있고 실패 모드 누락 아닌가? 새 분기마다 테스트?
4. **사이드이펙트** — 공용 유틸·서비스 변경이 다른 호출자에 영향?
5. **문서화** — 외부 사용자에게 영향을 주는 인터페이스 변경인데 README / CHANGELOG / docs 누락?

---

## 주의

- 요구사항이 애매하면 코드보다 먼저 사용자에게 질문 — 잘못된 가정으로 짠 PR 은 회수 비용이 크다.
- 미래 가설 (혹시 나중에 …) 으로 추상화 만들지 말 것. 호출자가 1곳이면 인라인이 정답.
- **테스트는 "새 기능이 기대대로 동작하는가" 중심** — 회귀 방지는 부수효과지 목표가 아님.
