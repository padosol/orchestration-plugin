# Bug 워크플로우 (버그 픽스)

**목적**: 기대 동작과 실제 동작의 차이를 제거.

**핵심 질문**: "왜 기대와 다르게 동작하는가?"

**최소 변경 원칙** — 무관한 리팩토링 / 스타일 변경 / 추가 기능 섞기 금지. 한 PR = 한 버그.

---

## Phase 템플릿

1. **재현** — 최소 재현 케이스 (`failing test` 또는 명령 시퀀스) 확보. **재현 안 되면 여기서 멈추고 정보를 더 수집** — "가끔 깨진다" 로 fix 시작 금지.
2. **기대 vs 실제** — 두 동작을 명확한 문장으로 기술. 기대 동작의 근거 (스펙 / 문서 / 기존 의도) 도 같이 기록.
3. **영향 범위** — 동일 코드 경로의 다른 호출자 / 영향받는 사용자 / 데이터 손상 가능성. hotfix 필요한지 판단.
4. **원인 추적** — 증상 → 원인으로 거슬러 올라가기. 가설 → 검증 사이클. 잘못된 가정을 명시 (예: "X 가 항상 non-null 이라 가정했으나 …").
5. **최소 수정** — **원인 자체를 고침**. 증상만 가리는 패치 금지:
   - try/except 로 예외 삼키기 ❌
   - 기본값으로 무마 ❌
   - 분기로 우회 ❌ (단, 빈번한 케이스를 빠르게 막아야 할 때는 hotfix 로 분리하고 후속 이슈 등록)
6. **회귀 테스트** — 1단계의 재현 케이스를 영구 테스트로 추가. **이 테스트는 fix 없으면 실패해야 함** — 빨강을 한 번 확인 후 fix 적용.
7. **유사 버그 점검** — 같은 패턴이 다른 곳에도 있는지. 예: 같은 null 체크 누락 / 같은 race condition 패턴 / 같은 입력 검증 누락.
8. **배포 후 모니터링** — 재발 감지 신호 (로그·메트릭·알람) 가 살아있는지 확인. 없으면 추가 검토.

---

## Review 체크리스트 (reviewer 용)

reviewer 가 PR 을 평가할 때 다음 항목을 우선:

1. **재현 케이스가 PR 에 포함됐는가** — 실패 테스트 또는 재현 시퀀스 없이 fix 만 있으면 needs-changes.
2. **회귀 테스트** — 추가된 테스트가 fix 없이는 실패하는 형태인가? assertion 이 너무 느슨하지 않은가?
3. **최소 변경** — diff 가 버그와 무관한 리팩토링 / 스타일 / 추가 기능 섞여 있지 않은가? 섞여 있으면 PR 분리 요구.
4. **증상 vs 원인** — try/except / 기본값 / 분기 우회로 증상만 가린 게 아닌가? 원인 설명이 PR 본문에 명확한가?
5. **동일 패턴 다른 곳** — 같은 종류 버그가 다른 모듈에도 있을 가능성 점검됐는가?

---

## 주의

- **재현 없이 수정 금지** — "고친 것 같다" 는 추측이고 재발 가능성 큼.
- 증상만 가리면 버그가 위장된 채 본 영역에서 회수 비용이 더 커짐.
- **테스트는 "같은 문제가 다시 안 생기는가" 중심** — 새 기능 추가 아님.
- 무관한 변경을 같이 넣지 말 것. 회수 / blame / 회귀 진단 모두 어려워진다.
